## 3. Iceberg Coupling and Merge-candidate Files

### 3.1 Iceberg coupling — hidden dependencies

**Definition:** coupling to private (`_`-prefixed) symbols, or to a module's implementation details rather than a declared API.

#### Case A — `generate/export.py` ← `lint_cmd` + `cite_cmd`

```python
# generate/export.py line 172
from src.cli.lint_cmd import _run_lint          # private

# generate/export.py line 211
from src.cli.cite_cmd import _MAPPING_PATH, _filter_applicable, _load_mapping  # all private
```

**Problem:** `generate.export` (the generation pipeline's export stage) calls into the
private implementation of two independent CLI commands.  Any refactor of `lint_cmd` or
`cite_cmd` silently breaks generate.  The coupling is *invisible* in `lint_cmd`'s and
`cite_cmd`'s own public interface.

**Remedy:** Extract `_run_lint`, `_MAPPING_PATH`, `_filter_applicable`, `_load_mapping`
into a shared service module (e.g., `src/cli/shared/lint_service.py` and
`src/cli/shared/cite_service.py`) that both the command and the pipeline can import.

#### Case B — `kb/rebuild.py` ← `verify_cmd`

```python
# kb/rebuild.py line 6
from src.cli.verify_cmd import collect_citation_verification_checks, render_citation_verification_results
```

**Problem:** KB rebuild borrows two functions from the `verify` CLI command.  The `verify`
command is registered as a direct top-level command — its internals should not be a
library for other commands.

**Remedy:** Move those two functions to `src/cli/shared/verify_service.py` (or into
`src/knowledge/`) and have both `verify_cmd` and `kb/rebuild.py` import from there.

#### Case C — `generate/__init__` → `history`

```python
# generate/__init__.py line 21
from src.cli.history import append_record
```

**Problem:** The generate command writes directly to the history log by importing from
the `history` CLI group.  This means `generate` is implicitly coupled to history's
storage schema.

**Remedy:** Move `append_record` to a lower-level service (e.g., `src/core/history_store.py`)
so both `history` and `generate` import the service, not each other's CLI layer.

## 4. Conclusion — Is fat-rotate v3 Granularity Governance Needed?

**Yes.** The following three issues independently justify a governance pass:

### 4.1 utils.py is a God Object

`utils.py` (310 lines) is imported by **26 of 80 files** (32.5%).  A blast radius this
wide means any change — even a docstring edit that triggers a linter — touches more than
a third of the module.

**Action:** Split `utils.py` into at least three domain-specific helpers:
- `utils_io.py` — file read/write, path helpers
- `utils_display.py` — Rich console, table, progress-bar helpers
- `utils_text.py` — text normalisation, encoding helpers

### 4.2 Iceberg Coupling Breaks Command Isolation

Three cross-command import pairs use private symbols, violating the principle that CLI
commands are isolated vertical slices:

```
generate.export  ──[private]──> lint_cmd._run_lint
generate.export  ──[private]──> cite_cmd._MAPPING_PATH / _filter_applicable / _load_mapping
kb.rebuild       ──[public fn]─> verify_cmd.collect_… / render_…
```

Until these are extracted to `src/cli/shared/` (or `src/core/`) services, renaming or
refactoring `lint_cmd`, `cite_cmd`, or `verify_cmd` silently breaks unrelated commands.

### 4.3 Size Distribution is Bimodal

The codebase simultaneously has files that are too large (9 files ≥ 250 lines) and files
that are too small (9 files < 50 lines).  The large files resist review and test
isolation; the small files add navigation overhead with no architectural gain.

**Recommended governance thresholds for src/cli/:**

| Signal | Threshold | Action |
|---|---|---|
| File too large | > 300 lines | Extract subpackage (model: `generate/`, `kb/`) |
| File too small | < 40 lines | Merge into nearest sibling |
| Cross-command import | any `_`-prefixed symbol | Move to `src/cli/shared/` service |
| Fan-in on single helper | > 10 importers | Split helper by domain |

### 4.4 What is *not* a problem

- **Within-group relative imports** (`kb.*` ↔ `kb._shared`, `history.*` ↔ `history._shared`) are
  healthy: they follow the same subpackage pattern as `generate/pipeline/`.
- **wizard → generate** and **workflow → generate** orchestration imports use the
  public `generate()` function — no private symbols, no iceberg risk.
- The `config` group's three-file split (`config_tools` + fetch_impl + mutations_impl)
  is a good example of the governance pattern already in practice.
