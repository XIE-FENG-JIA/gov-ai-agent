

# 產品 PRD：真實公文 AI Agent（暫定）

## 0. 產品一句話定位

打造一套「以真實台灣政府公文為黃金標準、由多個 AI Agent 互相審查監督」的公文撰擬系統，協助承辦人與公文寫作學習者，在數分鐘內產出符合《文書處理手冊》與「政府文書格式參考規範」的高品質公文草稿。([電子郵件政府][1])

---

## 1. 背景與問題

### 1.1 傳統公文撰寫的痛點

* 台灣政府公文有嚴格的格式與法規依據，例如公文類型（令、函、公告、書函、開會通知單、簽等）、用紙、字型、邊界、行距、標題層級等，均有明確規定。([主題網站][2])
* 文字用語也需符合慣用公文語氣，避免口語、不當稱謂與錯誤用字，許多機關會再自訂細部準則與常用範本。([電子郵件政府][3])
* 新進承辦人往往需要：

  * 花時間找「之前類似的公文」做參考
  * 一邊翻《文書處理手冊》一邊寫
  * 交給主管/同事反覆改才達標

### 1.2 一般生成式 AI 的問題

* 直接用公開 LLM 生成公文：

  * 容易**格式不符標準**（主旨、說明、辦法位置、字級、序號等不正確）
  * 可能**胡亂引用法條或程序**（幻覺問題）
  * 用語不符合機關「慣用文風」，甚至太口語或不夠謹慎
* 市面已有針對公文的 AI 助理，可協助生成草稿、推薦例稿、分類號與詞彙校正，但多以「單一 AI 助理」的使用者體驗為主，強調效率，較少將「可稽核的真實性、高準確度」做成清楚的產品承諾與多重防呆機制。([綠意拼圖][4])

### 1.3 本產品要解決的核心問題

> **如何在使用生成式 AI 的效率優勢下，仍能產出「接近資深承辦人審閱後成果」等級的台灣政府公文草稿，並且讓品質是「可檢查、可追溯」的？**

---

## 2. 產品願景與目標

### 2.1 願景

成為「繁中主權 AI 公文幕僚」：不只是幫忙打字，而是幫使用者查範例、比對法規、對齊格式與文風，最後交出一份可以直接送主管初審的公文草稿。

### 2.2 具體目標（v1）

1. **品質目標**

   * ≥ 90% 由系統產出的草稿，在公文專責人員的人工檢查下，可在「一次修正內」達到可發文標準（只需微調用詞與事實細節，而非整段重寫）。
2. **效率目標**

   * 相較於完全手動撰擬，平均可節省承辦人撰寫時間至少 50%。([綠意拼圖][4])
3. **合規目標**

   * 格式符合《文書處理手冊》與「政府文書格式參考規範」定義的基本項目（用紙尺寸、字型字級、邊界、主旨/說明/附件欄位等）之正確率達 95% 以上。([電子郵件政府][1])
4. **真實性與可追溯**

   * 每一份 AI 草稿，關鍵內容（依據、法條、程序）都能追溯到實際公文或法規來源，並以引用列表呈現。([APMIC][5])

---

## 3. 目標使用者與場景

### 3.1 角色

1. **政府機關承辦人 / 約聘僱人員**

   * 日常處理大量「函」、「公告」、「開會通知」、「簽」、「報告」等公文。
2. **機關內部主管 / 文書審核人員**

   * 負責審閱承辦人草稿，確認內容與格式。
3. **公文寫作學習者（考生、實習生、學生）**

   * 需要大量練習公文題目，希望有真實範例與 AI 回饋。

### 3.2 典型使用情境

1. **承辦人收到指示，要發一份「補助核定通知函」**

   * 在系統輸入：案由、對象、依據法規、核定金額、撥款方式、附檔清單…
   * 系統搜尋過往相似公文與網路上的公開公文範例。([經濟部標準檢驗局][6])
   * AI 生成草稿並經多 Agent 互審後，給出「接近可發文」版本。
2. **需要比照其他機關既有做法與文風**

   * 使用者輸入：「比照臺北市政府過去公告 XX 的寫法」，系統找出相近公告，產生同風格草稿。([台北市政府][7])
3. **公文寫作練習與批改**

   * 學習者輸入題目與自己的草稿，讓系統依「真實公文標準」給出修正與評語。

---

## 4. 產品定位與差異化

### 4.1 核心價值主張

1. **真實公文導向**：以真實公文與官方格式規範作為黃金標準，而非「一般商業書信」。([主題網站][2])
2. **多 Agent 互審機制**：不是單一模型輸出，而是由多個專責 Agent 互相找錯、挑戰與修正。
3. **可追溯來源**：重要敘述都帶有來源標註（公文編號、法規條文、官方網站連結）。
4. **人機協作，而非全自動發文**：最終決策仍由人承辦/主管，AI 是「超強草稿與審稿助理」。

### 4.2 與一般公文 AI 的區別

* 不只「幫你寫完」，還會：

  * 問你不清楚的細節（例如：依據條文、決行機關）
  * 主動標出風險（例如：法規可能已修正、條文日期過舊）
  * 提供「審稿報告」：這份公文的格式與內容檢查結果。

---

## 5. 範圍（Scope）

### 5.1 MVP v1 涵蓋公文類型

以常見且結構清楚的類型為優先：([主題網站][2])

* 對外：

  * 「函」（含一般函覆、通知函）
  * 「公告」
  * 「開會通知單 / 會勘通知單」
* 對內：

  * 「簽」（下級機關對上級機關首長之簽）
  * 「報告」
* 學習模式：

  * 題目 → AI 生成標準參考稿 → 比對使用者稿 → 給評語

### 5.2 不在 v1 範圍（Out of Scope）

* 機密公文實際發文處理（但可提供「格式模擬」）([slc.moj.gov.tw][8])
* 跟行政流程系統整合的「簽核流程引擎」（例如會簽、分文流程）
* 專業法律見解取代律師 / 主管機關正式解釋（AI 只能提出「建議」，不能被視為法律意見）

---

## 6. 核心功能需求

### 6.1 需求理解與對話引導 Agent

**目標**：將使用者的自然語言需求，轉成結構化「公文需求規格」。

* 功能：

  * 透過問答方式釐清：

    * 公文類型（函、公告、簽…）
    * 發文機關/收文對象
    * 案由與主旨重點
    * 依據法規或上級指示（若有）
    * 辦理方式 / 辦法 / 附件
  * 自動建立「公文需求表單」供後續 Agent 使用。
* 輸出：

  * 結構化 JSON / 表格（主旨、說明點列、依據、附件清單…）。

### 6.2 公文檔案檢索與範例蒐集 Agent

**目標**：從網路公開來源與內部知識庫，找出與需求相近的真實公文與範本。

* 資料來源構想：

  * 各機關公開公文與範本（如技術部門、標準檢驗局等網站提供的公文範本）。([經濟部標準檢驗局][6])
  * 文書格式相關法規與手冊。([電子郵件政府][1])
* 功能：

  * 自然語言轉搜尋查詢（RAG 檢索）
  * 回傳前 N 篇最相似的公文（含連結、主旨、摘要）
  * 標註每篇範例的：

    * 機關
    * 發文日期
    * 類型（函 / 公告 / 簽…）
* 輸出：

  * 「範例公文清單」＋被選用作主要參考的 1~3 篇。

### 6.3 模板選擇與骨架生成 Agent

**目標**：根據需求與規範，自動生成符合政府格式的「公文骨架」。

* 功能：

  * 根據公文類型選用合適模板：

    * 主旨格式
    * 說明段落（條列式）
    * 辦法 / 辦理要點（如適用）
  * 對齊官方格式要求：

    * 公文必要欄位（主旨、說明、附件、速別、密等、發文字號、日期、受文者等）([主題網站][2])
  * 預先建立 Word / ODF / PDF 排版參數（字體、字級、行距、邊界），或輸出為可套用樣板的結構（而非單純純文字）。([主題網站][9])

### 6.4 內容草稿生成 Agent

**目標**：在骨架上填入內容，產出第一版草稿。

* 資料與風格來源：

  * 前一階段找出的公文範例
  * 機關自訂用語表與常用句型（未來可由機關提供）
* 功能：

  * 依主旨與說明點列，生成內文，每一段對應「說明第 X 點」等架構
  * 盡可能引用實際法規條文與公文依據，並在文末或附錄列出來源。([APMIC][5])

### 6.5 多 Agent 互審機制（產品核心）

多個專責 Agent 在系統內對同一份草稿進行「輪流審查與互評」，最後產出一份整合稿與審核報告。

**擬定角色：**

1. **格式稽核 Agent**

   * 檢查：

     * 是否符合指定公文類型的欄位結構
     * 條列序號、標題層級是否正確
     * 主旨字數是否過長／過短
   * 輸出：格式問題清單 + 建議修正。

2. **文風與用語 Agent**

   * 檢查：

     * 是否使用正式公文用語
     * 是否出現口語、情感化字眼
     * 程度副詞（如「應」、「得」、「不得」）是否使用恰當
   * 可支援「模仿某機關文風」模式（未來可 fine-tune 或建立機關詞庫）。([台北市政府][7])

3. **事實與法規檢查 Agent**

   * 根據先前檢索到的公文與法規資料：

     * 確認引用法條是否存在且未明顯過期
     * 標出不確定或可能有風險的敘述（例如「依 X 法第 Y 條」但查無該條）
   * 輸出：風險警示清單 + 建議修改句子（不直接假裝肯定）。

4. **一致性與邏輯 Agent**

   * 檢查：

     * 主旨、說明、辦法三者是否一致
     * 文中提到的金額、日期、名詞是否前後一致
     * 有無邏輯跳躍或程序順序錯置

5. **總編輯 Agent**

   * 蒐集上述 Agent 意見，對草稿進行修訂，產出：

     * 修訂後公文草稿（v_final）
     * 「審核紀錄」：列出哪些地方被改、哪個 Agent 提出、理由為何

> 多 Agent 流程採「多輪制」：
> v0 草稿 → Agent A 審查 → v1 → Agent B 審查 → v2 → … → 總編 Agent 彙整 → v_final

### 6.6 使用者互動與人工覆核

* 功能：

  * 顯示「AI 審稿報告」，讓使用者一眼看到：

    * 格式是否合格
    * 有哪些法規/事實風險
    * 建議優先修改的三個重點
  * 允許使用者：

    * 逐段接受/拒絕 AI 建議
    * 手動編輯內容後，再請多 Agent「重跑審稿」一輪
* 目標：

  * 把系統定位為「公文寫作 IDE」，而不是黑箱產文器。

### 6.7 匯出與串接

* 支援匯出格式：

  * Word（.docx）套用機關樣板
  * PDF 預覽
  * 純文字（供貼到現有電子公文系統）
* 未來串接構想：

  * 與既有電子公文系統 API 或匯入格式整合（例如：SPEED 類系統等），讓草稿可直接匯入。([vitalsesp.com][10])

## 7. 非功能性需求

1. **隱私與安全**

   * 對可能含個資的內容（例如：姓名、身分證字號、地址）提供自動偵測與遮罩選項。
   * 公文內容儲存與模型訓練要支持「機關自有環境」部署，不強迫上雲。

2. **可靠性**

   * AI 生成失敗、外部檢索失敗時，要有清楚錯誤提示與 fallback（例如只提供模板而不生成依據）。

3. **可擴充性**

   * Agent 需要可配置（加入/關閉某個 Agent、調整審查規則），以適應不同機關要求。

4. **可解釋性**

   * 系統對每一項重大修改，都能給出「為何」：引自哪條規範、哪份範例，減少使用者不信任感。([APMIC][5])

---

## 8. 資料、模型與合規考量

1. **資料來源**

   * 以公開政府資訊與公文範本為主（.gov.tw 網站公文範本、格式規範、公告等）。([經濟部標準檢驗局][6])
   * 機關可選擇提供「內部歷史公文」作為私有知識庫，但須先進行去識別化。

2. **版權與開放資料**

   * 需確認各網站之內容使用規範（多數政府資料依開放資料授權條款使用，仍需標示來源）。([電子郵件政府][1])

3. **模型策略**

   * v1 可使用現成中文 LLM + RAG。
   * 後續視情況：

     * 微調「公文語料」以穩定文風
     * 將「格式/規範檢查」部分轉為 rule-based + LLM hybrid，以提高可控性。

---

## 9. 成功指標（KPI）

1. **使用者效率**

   * 平均每件公文撰寫＋修稿時間減少幅度（目標 ≥ 50%）。([綠意拼圖][4])
2. **品質**

   * 人工抽驗格式錯誤率（目標 ≤ 5%）
   * 法規引用錯誤率（目標 ≤ 3%，且全部有風險告警）
3. **採用度**

   * 每月活躍承辦人數（MAU）
   * 單位內部導入後，實際使用率（至少 70% 公文有經過系統草擬或審稿）
4. **學習者成效（若有學習模式）**

   * 使用系統三個月後，模擬測驗公文題目平均分數提升幅度

---

## 10. Roadmap（初稿）

* **Phase 0：概念驗證（PoC）**

  * 單一公文類型（例如「一般函」）
  * 基礎 RAG + 單 Agent 草稿生成 + 簡單格式檢查

* **Phase 1：MVP**

  * 支援「函 / 公告 / 簽」三類
  * 引入多 Agent 審查機制
  * 提供 Word 匯出 + 審稿報告

* **Phase 2：擴展與客製**

  * 支援機關自訂用語、模板
  * 學習者練習模式（題目 → AI 評分與講解）
  * 與既有電子公文系統深度整合

* **Phase 3：高階功能**

  * 自動產生「決行條列」建議
  * 對跨機關規定提供「風險提醒」（例如不同部會規定衝突可能性）
