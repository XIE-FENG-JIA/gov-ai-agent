## 🎨 設計亮點

### 1. 創新的多 Agent 並行審查機制

**傳統模型**: 單一 AI → 單一輸出
```
Input → LLM → Output (容易出現幻覺、偏差)
```

**本系統**: 9 個專責 Agent 協作
```
Input → 5 個生成 Agent → 初稿 v0
              ↓
           【4 個審查 Agent 並行執行】
              ↓
           總編輯 Agent → 最終稿 v_final
```

**優勢**:
- ✅ 多角度檢查，減少漏洞
- ✅ 避免單一 Agent 的偏差
- ✅ 並行執行提升 2.5 倍速度
- ✅ 每個修改都有明確溯源

### 2. Claude Code + Gemini CLI 的完美協作

**之前的選擇**:
- 要麼只用 Claude Code（代碼複雜時分析不全）
- 要麼只用 Gemini（開發效率低）
- 要麼兩者都用（API 成本翻倍）

**新方案**:
```
Claude Code (主開發，80% 時間)
    ↓ 每個 Phase 結束
Gemini CLI (全局審查，20% 時間) ← 通過 Claude 調用，無額外成本
    ↓ 返回結構化建議
Claude Code 改進 ↻
```

**優勢**:
- ✅ 快速迭代 + 深度分析
- ✅ 零 API 浪費（Gemini 通過 Claude 調用）
- ✅ 各司其職，最佳組合
- ✅ 開發效率 + 代碼質量雙提升

### 3. CLI-First 而非 Web 應用

**傳統 Web 應用的問題**:
- 前後端分離複雜
- 需要服務器部署
- 維護成本高
- 機構數據上雲有顧慮

**CLI-First 的優勢**:
- ✅ 簡潔、易維護
- ✅ 機構可本地部署
- ✅ 支援 Unix 管道組合
- ✅ 完全離線工作
- ✅ 易於集成到現有流程

### 4. 混合知識庫方案

**支援三種場景**:
1. **完全本地** - 內置種子數據，無需聯網
2. **在線同步** - 定期爬取政府網站更新
3. **自訂擴展** - 機構可添加內部公文和詞庫

**優勢**:
- ✅ 靈活適應各種環境
- ✅ 數據隱私與實時性兼顧
- ✅ 機關可定製化

## 🛠️ 技術架構圖

### 系統全景

```
┌─────────────────────────────────────────────────────────┐
│  使用者界面層 (User Interface)                           │
│  • 承辦人：快速生成公文草稿                              │
│  • 學習者：公文範例和反饋                                │
│  • 主管：審稿和質量檢查                                  │
└─────────────────────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────────────────────┐
│  CLI 命令層 (CLI Commands)                              │
│  • generate        （一鍵生成）                          │
│  • ask-requirements (需求分析)                          │
│  • search-examples  (範例檢索)                          │
│  • review-multi    (多 Agent 審查)                      │
└─────────────────────────────────────────────────────────┘
           ↓
┌──────────────────────────────────────────────────────────┐
│  Agent 核心層 (Agent Core System)                        │
│                                                          │
│  【生成 Agent】         【審查 Agent】                   │
│  1. 需求理解           1. 格式稽核                      │
│  2. 範例檢索           2. 文風檢查                      │
│  3. 骨架生成           3. 事實驗證                      │
│  4. 內容生成           4. 一致性檢查                    │
│  5. 總編輯            ↓ 並行執行                        │
└──────────────────────────────────────────────────────────┘
           ↓
┌──────────────────────────────────────────────────────────┐
│  基礎設施層 (Infrastructure)                             │
│  • LLM 後端: Ollama (本地) + Gemini (可選)              │
│  • 知識庫: SQLite + Chroma 向量 DB                      │
│  • 文檔處理: python-docx + markdown                     │
│  • 配置管理: YAML                                        │
└──────────────────────────────────────────────────────────┘
```

## 💬 關鍵決策和理由

### 決策 1: 為什麼選擇 CLI-First？
| 考量因素 | CLI-First | Web 應用 |
|---------|----------|--------|
| 部署複雜度 | ✅ 簡單 | ❌ 複雜 |
| 機構部署 | ✅ 本地部署 | ❌ 需要服務器 |
| 開發成本 | ✅ 低 | ❌ 高 |
| 易用性 | ✅ 易集成 | ✅ 易操作 |
| 管道組合 | ✅ 支援 | ❌ 不支援 |

**選擇 CLI-First 的原因**: 最大化靈活性和易部署性，同時保持簡潔性

### 決策 2: 為什麼 Agent 采用並行而非順序？
| 對比 | 並行審查 | 順序審查 |
|------|--------|--------|
| 耗時 | < 30 秒 | > 60 秒 |
| 漏洞率 | ✅ 低 | ❌ 高 |
| 複雜度 | ✓ 可控 | ✓ 簡單 |
| 多角度檢查 | ✅ 完全 | ❌ 單一視角 |

**選擇並行的原因**: 效率和質量的最佳平衡

### 決策 3: 為什麼 Claude + Gemini 協作？
| 對比 | 只用 Claude | 只用 Gemini | Claude + Gemini |
|------|----------|----------|----------------|
| 迭代速度 | ✅ 快 | ❌ 慢 | ✅ 快 |
| 全局分析 | ❌ 受限 | ✅ 完整 | ✅ 完整 |
| API 成本 | ✅ 低 | ✅ 低 | ✅ 零 |
| 開發效率 | ✅ 高 | ❌ 低 | ✅ 最高 |

**選擇 Claude + Gemini 的原因**: 完美組合快速迭代和深度分析，零 API 成本
